Sizing a Cluster for Your Workflow

This page answers: given workflow code, what does a Cadence cluster need to run it?

The Capacity Estimation page explains the formulas in detail. This page applies them to three real workflows from cadence-samples, each covering a different traffic and runtime pattern.

Each example follows the same steps:

  1. Read the workflow code and count event types.
  2. Derive traffic: workflows per day and average runtime determine peak concurrency and throughput.
  3. Plug the inputs into the estimator and read the full resource breakdown.

Example 1: Hello World​

The Hello World workflow is the simplest possible case: start, run one activity, finish.

func HelloWorldWorkflow(ctx workflow.Context, input sampleInput) (string, error) {
    ao := workflow.ActivityOptions{
        ScheduleToStartTimeout: time.Minute,
        StartToCloseTimeout:    time.Minute,
    }
    ctx = workflow.WithActivityOptions(ctx, ao)

    var greetingMsg string
    err := workflow.ExecuteActivity(ctx, HelloWorldActivity, input).Get(ctx, &greetingMsg)
    if err != nil {
        return "", err
    }
    return greetingMsg, nil
}

Example 2: Expense Approval​

The expense workflow models a human-in-the-loop approval process: create a report, wait for a manager to approve, process payment. It has three activities and a long average runtime because the workflow idles while a human reviews the request.

func sampleExpenseWorkflow(ctx workflow.Context, expenseID string) (string, error) {
    // Activity 1: create the expense report
    err = workflow.ExecuteActivity(ctx1, createExpenseActivity, expenseID).Get(ctx1, nil)
    if err != nil {
        return "", err
    }

    // Activity 2: block until a human approves or rejects (could take hours or days)
    var status string
    err = workflow.ExecuteActivity(ctx2, waitForDecisionActivity, expenseID).Get(ctx2, &status)
    if err != nil {
        return "", err
    }

    if status != "APPROVED" {
        return "", nil
    }

    // Activity 3: process payment
    err = workflow.ExecuteActivity(ctx2, paymentActivity, expenseID).Get(ctx2, nil)
    return "COMPLETED", nil
}

Example 3: Order Processing​

This is the scenario pre-loaded in the estimator. It covers a typical e-commerce fulfillment flow: validate an order, fulfill it, send a confirmation — with a cancellation signal, a fulfillment timeout, and a search attribute update for order tracking.

func OrderProcessingWorkflow(ctx workflow.Context, order OrderInput) error {
    // Mark order as in-progress in the visibility index
    _ = workflow.UpsertSearchAttributes(ctx, map[string]interface{}{
        "OrderStatus": "processing",
    })

    // Timer: escalate if not fulfilled within 30 minutes
    timerCtx, cancelTimer := workflow.WithCancel(ctx)
    _ = workflow.NewTimer(timerCtx, 30*time.Minute)

    // Signal: listen for customer cancellation
    cancelCh := workflow.GetSignalChannel(ctx, "cancel-order")
    _ = cancelCh

    ao := workflow.ActivityOptions{StartToCloseTimeout: 10 * time.Minute}
    actCtx := workflow.WithActivityOptions(ctx, ao)

    // Activity 1: validate
    if err := workflow.ExecuteActivity(actCtx, ValidateOrder, order).Get(actCtx, nil); err != nil {
        return err
    }

    // Activity 2: pick, pack, ship
    if err := workflow.ExecuteActivity(actCtx, FulfillOrder, order).Get(actCtx, nil); err != nil {
        return err
    }

    cancelTimer()

    // Activity 3: send confirmation email
    if err := workflow.ExecuteActivity(actCtx, SendConfirmation, order).Get(actCtx, nil); err != nil {
        return err
    }

    return nil
}

Resource estimates are derived from observed production ratios at ~11,444 updates/sec. They are order-of-magnitude guides for initial cluster planning, not production sizing guarantees. Small deployments (<100 updates/sec) have higher per-update overhead due to fixed service minimums; plan for at least 1 instance per service regardless of computed values. For multi-cluster or global domain setups, multiply storage and Cassandra node count by the number of replication targets.

Cassandra utilization​

The resource estimates above show allocated capacity. Utilization is how much of that allocation the cluster is actually consuming at peak.

MetricHealthy rangeAction if outside range
CPU utilization per node20–70%Below 10%: overprovisioned; above 80%: add nodes or reduce compaction pressure
Memory utilization60–80% of node RAMCassandra uses memory for read/write caches and heap; above 85% risks GC pauses
Disk utilizationBelow 60%Cassandra needs headroom for compaction; above 70% risks write stalls

At the throughput levels in examples 1 and 2 (under 1 update/sec), utilization will appear artificially low because you are running the 3-node production minimum with near-zero traffic. That is expected: the minimum exists for fault tolerance (RF=3), not throughput. Utilization numbers become meaningful above roughly 100 updates/sec where load drives resource consumption rather than fixed minimums.

Usage vs. allocation. When comparing environments, use the ratio of actual CPU consumed (from Grafana) to allocated CPU (replicas x cores/pod). A ratio below 15% on History or Cassandra with healthy traffic suggests the cluster was provisioned for a peak that never materialized. The reference cluster spreadsheet uses this ratio to flag overprovisioned environments.

Worker sizing​

The estimator does not size workers because worker resource requirements depend on what your activities do, not on Cadence server throughput. Two steps:

Step 1: count concurrent activities at peak.

concurrentActivities = workflowsAtPeak x activitiesPerWorkflow

Step 2: apply a core-per-activity ratio based on your activity profile.

Activity profileStarting ratioExample
I/O-bound (network calls, DB queries, waiting on humans)1 core per 50 concurrentexpense waitForDecision
Mixed (I/O + moderate CPU)1 core per 25 concurrentorder processing
CPU-bound (encoding, ML inference, hashing)1 core per 10 concurrentimage processing, compression

Validate by watching worker CPU utilization in Grafana after launch. If sustained above 80%, add instances. If below 20%, reduce. See the worker configuration guide for polling concurrency tuning.

Reading the outputs​

A few checks before treating the estimates as final:

  • actionsPerSec below 100: fixed service minimums dominate. Plan for at least one instance of each service regardless of the linear model output.
  • totalRetained larger than expected: review retention settings. At Uber, production domains typically use 7–30 days. For workflows older than 30 days, enable archival to move closed history to blob storage.
  • totalActions higher than expected: check heartbeat frequency. Each heartbeat writes one record. An activity heartbeating every 5 seconds on a 1-hour task adds 720 records; that may be intentional (progress tracking) or accidental (too-frequent polling).
  • Storage growing faster than the formula predicts: payload sizes in production are often larger than estimated. Use p95 or p99 sizes from your serialized workflow inputs in Grafana rather than the average.
